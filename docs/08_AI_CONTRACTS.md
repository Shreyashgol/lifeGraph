# AI Contracts

## Project

**LifeGraph — AI-Powered Personal Intelligence Engine**

**Version:** 1.0

**Status:** Canonical AI Reasoning Contract

---

# 1. Purpose

This document defines the responsibilities, boundaries, inputs, outputs, validation requirements, and failure handling for every AI reasoning component within LifeGraph.

The AI layer is responsible for reasoning only.

It is **not** responsible for:

* Database access
* State mutation
* Business rules
* Persistence
* API responses

The AI proposes.

The application decides.

---

# 2. AI Philosophy

LifeGraph treats Large Language Models as **reasoning engines**, not application engines.

Every AI component follows the same lifecycle.

```text
Business Input
      │
      ▼
Prompt Builder
      │
      ▼
LLM
      │
      ▼
Structured Proposal
      │
      ▼
Schema Validation
      │
      ▼
Business Validation
      │
      ▼
Approved Result
```

No AI output should directly modify application state.

---

# 3. Core Design Principles

Every AI component must satisfy the following principles.

### Principle 1 — One Responsibility

Each AI service performs exactly one reasoning task.

Examples:

* Activity Understanding
* Memory Proposal
* Behaviour Analysis
* Insight Generation
* Recommendation Generation
* Summary Generation

Never combine multiple reasoning objectives into one prompt.

---

### Principle 2 — Structured Outputs

Every AI service must return structured JSON.

Long-form natural language is permitted only for the Summary service.

---

### Principle 3 — Deterministic Integration

Although LLMs are probabilistic, their integration into the application must be deterministic.

Every response must pass:

* Schema validation
* Business validation
* Confidence evaluation

before becoming part of the system.

---

### Principle 4 — Explainability

Every conclusion generated by AI must include supporting reasoning or evidence references.

The system must never present unexplained recommendations.

---

### Principle 5 — Replaceable Providers

AI contracts must remain independent of the model provider.

The same contract should work with:

* Groq
* OpenAI
* Anthropic
* Gemini
* Local models

No provider-specific assumptions should appear in the contract.

# 5. Common AI Service Contract

Every intelligence service should conceptually implement the following interface.

```python
class IntelligenceService:

    async def reason(
        self,
        context: AIContext
    ) -> AIProposal:
        ...
```

Responsibilities:

* Build prompt
* Invoke LLM
* Parse response
* Validate schema
* Return proposal

The service should never:

* Save data
* Update GraphState
* Call repositories
* Perform business decisions

## Output

## Responsibilities

The service should answer:

* What kind of work is this?
* Which project is involved?

* Infer long-term behaviour

# 9. Confidence Contract

Every AI proposal must include a confidence score.

Range:

```text
0.00 → 1.00
```

Suggested interpretation:

| Confidence | Meaning                         |
| ---------- | ------------------------------- |
| 0.90–1.00  | High confidence                 |
| 0.75–0.89  | Acceptable                      |
| 0.60–0.74  | Requires caution                |
| Below 0.60 | Reject or request clarification |

Confidence alone should never determine acceptance.

Business rules remain authoritative.

# 10. AI Failure Contract

Recoverable failures:

* Timeout
* Temporary provider error
* Invalid JSON
* Rate limiting

Strategy:

```text
Attempt

↓

Retry

↓

Retry

↓

Fallback

↓

Return Failure
```

Non-recoverable failures:

* Invalid input
* Missing required context
* Unsupported schema

These should fail immediately with clear diagnostics.

# Part 1 Summary

The AI layer exists to produce **structured proposals**, not final decisions.

Every proposal must:

* Solve one reasoning task.
* Produce structured output.
* Include confidence.
* Remain explainable.
* Be validated before entering the application.

This separation ensures that AI remains a reasoning component while the application retains authority over business decisions.

# 14a. Evaluation Intelligence Contract

## Purpose

Judge the quality of a structured proposal produced by another service
(LLM-as-a-judge). Evaluation does **not** re-solve the task; it decides what
should happen to a proposal.

Distinct from Reflection: Reflection is holistic, end-of-graph QA; Evaluation is
per-proposal quality control positioned immediately after the producing node.

## Input

Receives:

* The original user input (e.g. the raw activity)
* The structured proposal produced by another service
* Relevant context

## Output

Returns an `EvaluationDecision`:

* decision — `approve | retry | reject`
* score — 0.0–1.0
* feedback — human-readable assessment
* retry_reason — specific, actionable correction (empty when approving)

## Responsibilities

* Assess correctness (category, project, duration) and whether confidence
  matches the evidence.
* Emit actionable feedback so a retry has concrete guidance.
* Never persist, never mutate state, never output a corrected proposal.

## Retry Semantics

Two independent retry mechanisms exist:

* **Transport/parse retry** — mechanical, owned by the intelligence service
  (timeouts, invalid JSON, schema-invalid responses); bounded by the prompt's
  `retry_limit`.
* **Quality retry** — the Evaluation decision. The **graph** owns this loop via a
  conditional edge (Evaluation → producing node with `retry_reason` injected),
  capped at **2** quality-retries before the proposal is rejected.

Pydantic Validation

# 18. Retry Contract

Retries should be controlled and observable.

Recommended policy:

| Failure Type             |                    Retry |
| ------------------------ | -----------------------: |
| Timeout                  |                      Yes |
| Temporary provider error |                      Yes |
| Invalid JSON             |                      Yes |
| Rate limiting            |                      Yes |
| Schema mismatch          | Once after prompt repair |
| Invalid business rules   |                       No |

Maximum retries:

```text id="vrru6t"
3
```

After the limit, return a structured failure.

# 20. AI Safety Contract

The AI layer must follow strict safety rules.

Never:

* Invent user history.
* Modify memory without evidence.
* Present assumptions as facts.
* Produce generic recommendations without personalization.
* Expose internal prompts.
* Leak API keys or internal metadata.

If required information is missing, prefer uncertainty over fabrication.

# 23. Service Independence Contract

Every intelligence service must remain independent.

Services should never call each other.

Example:

```text id="r1ix7o"
Activity Service

✗

calls

Memory Service
```

Instead:

```text id="s3feyz"
Activity Service

↓

Graph State

↓

Memory Service
```

The graph coordinates services.

Services never coordinate themselves.

---

# 24. AI Contract Summary

Every intelligence component in LifeGraph follows the same contract:

1. Receive structured context.
2. Solve one reasoning problem.
3. Produce a structured proposal.
4. Include confidence.
5. Pass schema validation.
6. Pass business validation.
7. Return to the graph.
8. Never persist directly.

This architecture cleanly separates probabilistic reasoning from deterministic application logic.

---

# 25. Engineering Principles

The AI layer is governed by these principles:

* One service, one responsibility.
* One prompt, one reasoning task.
* Structured outputs by default.
* Explainability over opacity.
* Validation before persistence.
* Provider independence.
* Observability by design.
* Business logic outside the LLM.

Following these principles ensures that LifeGraph remains maintainable, trustworthy, and extensible as both the product and underlying AI models evolve.
